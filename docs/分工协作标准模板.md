# 分工协作标准模板（短版）

适用场景：多人并行开发（A/B/C 分层、共享配置、跨模块联调）。

## 1) Owner 规则

- 每个共享文件仅 1 个 owner（配置文件必须明确 owner）。
- 非 owner 不直接改共享文件，只通过 PR review 提建议。

## 2) 契约冻结

- 开发前冻结：字段名、枚举、默认值、阈值来源、schema 版本。
- 任何契约变更先提“接口变更提案”，获批后再改代码。

## 3) 分支基线

- 每日开始前执行：`git fetch origin && git rebase origin/main`。
- 提 PR 前必须再次对齐 `main`。

## 4) PR 前门禁（强制）

- 边界检查：不得修改非本成员授权文件。
- 契约检查：字段、命名、必填、版本一致。
- 测试检查：本成员必测用例全绿。

## 5) 评审边界

- 主审：判定是否可合并。
- 副审：补风险与兼容性。
- 每条评审意见必须包含：问题点、影响范围、严重级别、修改建议。

## 6) 合并条件

- 无 BLOCKER。
- 无未解决 MAJOR。
- CI 绿 + 指定验收脚本通过。
- 放行条件复选框全部勾满。

## 7) 节奏控制

- 小 PR、快评审、快回归。
- 每波次做一次最小联调，不等全量开发完成。

## 8) 常见错误预防

### 8.1 运行产物入库

**问题**：logs/*.pid、data/*.db、__pycache__/ 等文件被 commit

**预防措施**：
- 项目初始化时立即配置完整的 `.gitignore`
- 每次 `git add -A` 前用 `git status` 检查要提交的文件
- 特别注意：任何程序运行后生成的文件都可能污染仓库

**正确流程**：
```bash
# 运行程序
python3 scripts/xxx.py
# 检查状态
git status
# 只添加代码文件，手动排除产物
git add scripts/ configs/ tests/
# 或者使用 interactive add
git add -i
```

### 8.2 CI 绕过

**问题**：测试失败时用跳过/简化 CI 方式"绕过"

**预防措施**：
- 必须修复根因，禁止绕过 CI
- 如遇超时/环境问题，提出来集体讨论解决方案
- CI 改弱化必须说明理由并存档

**正确心态**：
- ❌ 改 `--self-only` 跳过检查 → 假通过
- ✅ 修复测试超时/修复测试本身 → 真通过

### 8.3 缺少变更说明

**问题**：代码改了但 PR 描述没说为什么改、有什么影响

**预防措施**：
- PR 模板必须包含：变更理由、影响范围、回滚方案
- 重要配置变更（provider/model/阈值）必须额外说明
- 涉及其他成员的文件时，必须 @相关人确认

**PR 描述必填项**：
```
## 变更理由
- 为什么改？

## 影响范围
- 哪些模块/功能受影响？

## 回滚方案
- 如何快速回退到上一版本？
```

### 8.4 代码变更后不测试

**问题**：改了代码没本地测试就提 PR

**预防措施**：
- 提 PR 前必须运行：本模块相关测试
- 涉及共享配置时，必须验证端到端场景
- 用 `git diff` 检查改动了什么，再决定测什么
