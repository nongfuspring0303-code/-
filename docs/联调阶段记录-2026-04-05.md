#### 记忆系统维护任务
- **任务ID**: `memory-maintenance-weekly`
- **执行时间**: 每周六 10:00（`0 10 * * 6`）
- **执行内容**: 运行记忆维护脚本，清理+一致性检查+索引重建
- **配置位置**: `data/raw/stock-monitor-cron-config.json`

### 9.2 定时任务机制说明

定时任务基于 **OpenClaw Gateway**（端口18789）：
- 配置文件: `stock-monitor-cron-config.json`（随代码分发）
- Gateway 加载配置后按时间触发任务
- 任务在 main 会话中执行（类似 heartbeat）
- 拉取代码后 Gateway 会自动加载新配置

### 9.3 对比 WorldMonitor

| 特性 | EDT项目（OpenClaw） | WorldMonitor |
|------|-------------------|---------------|
| 定时任务 | ✅ 内置 Gateway Cron | ❌ 依赖 Railway |
| 自动触发 | ✅ 配置后自动执行 | 需额外部署 Railway |
| 代码包含 | ✅ 在 cron-config.json | ❌ 需独立部署 |

## 10. 实时新闻监控功能（2026-04-05 补充）

### 10.1 新增功能

#### 实时新闻监控脚本
- **文件**: `scripts/realtime_news_monitor.py`
- **功能**:
  1. 持续轮询新闻源（默认60秒间隔）
  2. 检测新新闻后立即触发 EventCapture
  3. 触发后立即运行 A/B 计算并推送到 C 模块

#### 使用方式
```bash
# 启动实时监控（后台运行）
cd <LOCAL_WORKSPACE>/事件驱动交易模块阶段二
nohup python3 scripts/realtime_news_monitor.py --poll-interval 60 --api http://127.0.0.1:8787 > logs/realtime_news_monitor.log 2>&1 &

# 测试模式（只打印新闻，不触发A/B计算）
python3 scripts/realtime_news_monitor.py --once --test

# 单次检查
python3 scripts/realtime_news_monitor.py --once
```

#### 定时启动任务
- **任务ID**: `realtime-news-monitor-startup`
- **执行时间**: 工作日 9:00（`0 9 * * 1-5`）
- **功能**: 检查并启动实时监控进程

### 10.2 监控模式对比

| 模式 | 触发条件 | 说明 |
|------|---------|------|
| **定时批量** | 每天3次（9:30,12:00,15:00 ET） | 批量计算+推送 |
| **实时监控** | 新闻到达立即触发 | 持续运行，实时响应 |

### 10.3 工作流程

```
新闻源 → realtime_news_monitor.py (轮询)
         ↓
    EventCapture (关键词/VIX检测)
         ↓
    触发条件? → FullWorkflowRunner (A/B计算)
         ↓
    sector_update → C模块API (/api/ingest/sector-update)
         ↓
    opportunity_update → C模块API (/api/ingest/opportunity-update)
```

### 11.1 实时新闻监控获取超时

**问题描述**：
- `realtime_news_monitor.py` 运行时，`DataAdapter.fetch_news()` 获取新闻超时
- 导致实时监控无法自动触发

**现象**：
- 手动触发 A/B 计算可以正常推送数据到 C 模块
- 实时监控自动轮询卡住，无法获取新闻
- 手动调用 `NewsIngestion().run()` 需要 **61秒**才返回

**诊断结果**（2026-04-06）：
- 导入模块：0.1秒 ✅
- 实例创建：0.0秒 ✅
- NewsIngestion.run()：**61秒** ❌
- 外部RSS源响应极慢，远超配置的8秒超时

**可能原因**：
1. 外部 RSS 新闻源响应极慢（61秒）
2. 超时配置（8秒）被重试机制绕过
3. 轮询间隔（30秒）短于获取耗时（61秒）

**当前状态**：未修复

**建议**：
1. 添加超时保护（缩短超时或直接用fallback）
2. 更换更快的新闻源
3. 延长轮询间隔
4. 当前手动触发方式可正常工作
