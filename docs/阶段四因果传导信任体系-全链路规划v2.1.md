# 阶段四因果传导信任体系（全链路规划 v2.1）

**版本**: 2.1  
**日期**: 2026-04-11  
**状态**: 已批准，待实施  
**适用范围**: 事件驱动交易模块阶段二（美股全板块）

---

## 1. 目标与原则

### 1.1 核心目标

将当前弱映射链路：

`新闻 -> 板块 -> 个股`

升级为可审计、可验证、可回放的强因果链路：

`事件对象 -> 冲击抽象 -> 因子向量 -> 资产验证 -> 板块排序 -> 个股暴露 -> 交易适配`

### 1.2 设计原则

- 先判冲击，再判板块，再选个股
- 禁止新闻直接跳股票
- 没把握不输出（NO_ACTION 优先）
- 输出数量硬限制，优先质量
- 所有关键决策必须可解释（含分数拆解和拒绝原因）

## 3. 全链路架构（L1-L6）

### L1 事件标准化层

- 输入: NewsIngestion / EventObjectifier
- 输出: 标准事件对象（schema v1.1）
- 责任: 保证 trace_id、news_timestamp、event_type、severity 完整
- 事件标签升级: 输出 `event_type_lv1` + `event_type_lv2`（二级标签作为实际映射主键）
- 双置信度: 输出 `classification_confidence` 与 `market_impact_confidence`

### L2 冲击抽象层（TM-1）

- 输出 `shock_profile`
- 至少区分: primary_shock / secondary_shock / narrative_shock
- 标准冲击原型（首版 12 类）:
  - demand_positive, demand_negative
  - supply_positive, supply_negative
  - inflationary_shock, disinflationary_shock
  - rates_up, rates_down
  - liquidity_tightening, liquidity_easing
  - risk_on, risk_off

### L3 因子向量层（TM-2）

- 输出 `raw_macro_factor_vector`（不含市场验证修正），范围 -100 到 +100
- 首版 9 因子:
  - risk_appetite
  - rates
  - inflation
  - growth
  - usd
  - liquidity
  - volatility
  - input_cost
  - credit_stress

### L4 资产验证层（TM-4）

- 输出 `asset_validation`
- 输出 `macro_factor_vector`（在 `raw_macro_factor_vector` 基础上施加市场验证修正）
- 验证篮子（首版 5 组）:
  - rates basket
  - risk-off basket
  - commodity basket
  - USD basket
  - growth-vs-value basket
- 闸门规则: 统一读取 `configs/gate_policy.yaml`（禁止在代码与文档中写死阈值）

### L5 板块与个股排序层（TM-5/TM-6）

- 板块输出:
  - primary_sector (<=1)
  - secondary_sector (<=1)
  - avoid_sector (<=2)
- 个股输出:
  - stock_candidates (<=5)
  - 仅允许来自 premium_stock_pool

### L6 交易适配层

- 输出行动: TRADE / WATCH / NO_ACTION
- 接入 RiskGatekeeper / PositionSizer / 执行层
- 记录拒绝原因与风控触发原因

## 5. 打分与分级规则

### 5.1 因子生成规则

L3 原始因子（成员A）：

`raw_factor = 基础模板 x 严重度系数 x 生命周期系数 x 新颖度系数 x 疲劳折扣 x 作用域修正`

L4 生效因子（成员B）：

`effective_factor = raw_factor x 市场验证修正`

建议系数：

- 严重度: E1=0.4, E2=0.7, E3=1.0, E4=1.35
- 生命周期: Detected=0.35, Verified=0.60, Active=1.00, Continuation=0.75, Exhaustion=0.40, Dead=0.10
- 疲劳折扣: `effective = raw x (1 - fatigue_final / 180)`
- 市场验证修正由 L4 计算并写回 `macro_factor_vector`

### 5.2 板块评分

`Final_Sector_Score = 0.45*Factor_Exposure + 0.25*Asset_Validation + 0.20*Fundamental_Path + 0.10*Narrative_Path - Penalties`

输出:

- top_positive_sectors
- top_negative_sectors
- mixed_or_unclear

### 5.3 个股评分

`Stock_Transmission_Score = 0.35*Event_Exposure + 0.25*Event_Relevance + 0.20*Relative_Strength + 0.10*Liquidity_Score + 0.10*Risk_Filter_Score`

补充说明：

- `Event_Exposure` 来源于 `ticker x event_type_lv2` 暴露度矩阵
- 暴露度矩阵为持续回验资产，按历史命中率滚动更新
- 个股仍只允许在 premium_stock_pool 内排序

个股分类:

- first_order_carriers
- leveraged_proxies
- contaminated_names

### 5.4 行动分级

- A 档: 输出 1 主板块 + 1-5 股票
- B 档: 仅输出主板块
- C 档: NO_ACTION + reason

分级阈值统一读取 `configs/gate_policy.yaml`:

- `signal_grade.a`
- `signal_grade.b`
- `path_dominance.min_gap`

## 10. 门禁与验收标准

### 10.1 允许进入交易主链的条件

同时满足 `configs/gate_policy.yaml.trade_admission` 定义的条件（字段保持如下）：

1. `A0`
2. `confidence`
3. `asset_validation.score`
4. `path_dominance.score`
5. `sector_top1_top2_gap`
6. `A0_5`
7. `fatigue_final`

说明：阈值数字不再在文档中重复维护，避免门槛漂移。

参数语义补充：

- `gate_policy.asset_validation.trade_min`：用于状态机第 3 步（Asset Validation Gate）前置过滤
- `gate_policy.trade_admission.asset_validation_min`：用于状态机第 5 步（Trade Admission Gate）最终准入

### 10.3 决策状态机优先级（唯一裁决顺序）

为避免同一事件出现 WATCH/TRADE 双结论，统一采用单一状态机，按以下顺序裁决：

1. **Data Integrity Gate**：关键字段缺失、时间戳非法、schema 不兼容 -> `NO_ACTION`
2. **Mixed Regime Gate**：`mixed_regime=true` 且未达到 override 条件 -> `WATCH`
3. **Asset Validation Gate**：未达到 `gate_policy.asset_validation.trade_min` -> `WATCH` 或 `NO_ACTION`
4. **Risk Gate**：流动性红区、风控红线、执行不可达 -> `NO_ACTION`
5. **Trade Admission Gate**：全部交易准入条件满足 -> `TRADE`
6. **Fallback Rule**：其余情况默认 `WATCH`

状态机枚举（输出字段 `state_machine_step` 统一使用以下 snake_case 值）：

- `data_integrity`
- `mixed_regime`
- `asset_validation`
- `risk_gate`
- `trade_admission`
- `fallback`

`mixed_regime` override 条件（首版配置化）：

- 读取 `gate_policy.mixed_regime_override.enabled`
- 当 `enabled=false` 时，不允许 override（`mixed_regime` 直接输出 `WATCH`）
- 当 `enabled=true` 时，需同时满足：
  - `asset_validation.score >= gate_policy.mixed_regime_override.asset_validation_min`
  - `path_dominance.score >= gate_policy.mixed_regime_override.path_dominance_min`
  - `sector_top1_top2_gap >= gate_policy.mixed_regime_override.sector_gap_min`

强制规则：

- 后置门不能覆盖前置门的拒绝结论
- 任何模块只允许输出“建议状态”，最终状态仅由状态机给出
- 状态机输出必须携带 `state_machine_step` 与 `gate_reason_code`

## 11. 风险与防补漏策略

### 11.1 主要风险

- 因子模板和暴露矩阵参数失真
- 资产验证滞后导致错过初段行情
- 冲突路径未裁决导致信号打架

### 11.2 防补漏策略

- 先冻结 schema 与配置接口，再写逻辑
- 每条输出必须落盘 score_breakdown + gate_reason
- 保留旧链路 fallback，但默认走新链路
- 先上 MVP（8 事件模板 / 9 因子 / 12 板块 / 5 资产篮子），逐步扩容

---

## 12. 最终定义

本规划不是“多给股票”的系统，而是“先验证因果链，再给少量高把握标的”的系统。  
在美股全板块场景下，系统默认保守输出：宁缺毋滥，优先可解释、可复核、可执行。

## 13. 外部制度文件吸收项（v1.1 对齐）

以下条目来自 `新闻源 -> 板块 -> 个股 自动映射制度文件 v1.1`，已并入本规划并作为默认执行标准：

### 13.1 输出与行为约束

- 默认 `NO_ACTION` 优先，低把握场景不强行输出
- 硬限制保持:
  - primary_sector <= 1
  - secondary_sector <= 1
  - stock_candidates <= 5
- A/B/C 分层保留:
  - A: 板块 + 1-5 只股票
  - B: 仅板块
  - C: NO_ACTION + reason

### 13.2 个股层筛选纪律

- 个股仅允许从 `configs/premium_stock_pool.yaml` 选取
- 禁止全市场直接扫票
- 个股层必须给出可审计评分拆解（各因子分项 + 风险扣分）
- 个股层必须记录 `event_type_lv2` 对应的 `event_exposure` 证据

### 13.3 评分与门槛（配置化）

- 采用四因子框架并保留可扩展接口:
  - event_relevance
  - relative_strength
  - liquidity
  - risk_filter
- A/B/C 阈值作为 `gate_policy.yaml` 初始默认值管理，禁止硬编码
- 当个股领先优势不足时，自动降级到 B 档（仅板块）

### 13.4 配置化与热更新要求

- 事件到冲击映射落地到 `configs/event_to_shock.yaml`
- 板块暴露矩阵落地到 `configs/sector_exposure_matrix.yaml`
- 由 ConfigCenter 统一加载，支持热更新
- 运行时保留规则版本号，确保回放时可复现

兼容策略：

- 允许 `event_to_sector_mapping.yaml` 作为只读兼容别名 1 个版本周期
- 新实现仅允许读取 `event_to_shock.yaml` + `sector_exposure_matrix.yaml`

### 13.5 可观测性与审计字段

- 输出必须携带:
  - `schema_version`
  - `trace_id`
  - `news_timestamp`
  - `gate_reason`
  - `score_breakdown`
- 每次降级或拒绝都必须记录明确 reason（便于复盘和调参）

### 13.6 与 v2.0 的衔接原则

- v1.1 的“新闻 -> 板块 -> 个股”规则作为执行层约束
- v2.0 新增的“冲击/因子/资产验证/路径裁决”作为上游因果解释层
- 两者关系: `v2.0 负责解释与验证`，`v1.1 负责输出纪律与数量控制`

### 14.1 TM-3 传导图谱协议（强制）

新增标准字段 `transmission_paths`，每个 path 结构必须包含：

- `path_id`: 唯一路径标识
- `path_type`: `fundamental | asset_pricing | narrative`
- `horizon`: `intraday | 1-5D | multiweek`
- `persistence`: `fast | medium | slow`
- `confidence`: 0-100
- `nodes`: 节点数组
- `edges`: 边数组

边结构统一为：

```json
{
  "from": "input_cost_up",
  "to": "airlines",
  "weight": -0.85,
  "lag": "fast",
  "path_type": "fundamental"
}
```

规则：

- 每次事件至少产出 3 条路径（fundamental/asset_pricing/narrative 各 1 条）
- 不允许只输出 narrative path 进入 TRADE
- path 缺边或缺节点时，自动降级 WATCH

### 14.2 资产节点白名单（TM-4 强约束）

固定资产节点集合（首版冻结，不允许运行时随意扩散）：

### 14.4 主路径裁决补充（定量化）

### 14.5 与现有模块兼容策略

## 15. 回测与防泄漏协议（必须执行）

### 15.1 时序定义

- `t_event`: 新闻事件原始发布时间
- `t_detect`: 系统检测到事件时间
- `t_decide`: 生成 TRADE/WATCH/NO_ACTION 的决策时间
- `t_execute`: 执行层下单时间

强约束：

- 任一特征只能使用 `<= t_decide` 可获得的数据
- 禁止使用 `t_decide` 后价格/成交信息参与当次决策评分
- `asset_validation` 的输入时间戳必须落盘并可审计

### 15.2 回测方法学

- 使用 walk-forward 时间切分
- 使用 purged/embargo 防止标签泄漏
- 必须纳入交易成本、滑点、流动性约束
- 必须输出分市场状态（regime）表现，不接受仅整体均值

### 15.3 Point-in-Time（PIT）快照强制项

回测与复盘必须使用决策时点可见数据，禁止使用“当前最新数据”回填历史。

必须快照并落盘：

- `feature_snapshot_at_t_decide`（全部特征原值）
- `rule_snapshot_version`（gate_policy、metric_dictionary、mapping 配置版本）
- `universe_snapshot`（当日可交易股票池与白名单）
- `asset_whitelist_snapshot`（资产节点白名单版本）

强制校验：

- 回放时若任一快照缺失，样本标记为 `INVALID_FOR_BACKTEST`
- 白名单与股票池必须支持“按日期切面”查询，不允许用当前版本回放历史
- 任何 PIT 违规样本不得用于绩效统计

### 15.4 稳健性门禁

- 阈值扰动（+-10% / +-20%）仍保持方向一致
- 权重扰动后绩效不出现断崖
- 子样本（高波动/低波动）表现不可失真
- 任一稳健性门禁失败 -> 自动降级 WATCH/NO_ACTION
