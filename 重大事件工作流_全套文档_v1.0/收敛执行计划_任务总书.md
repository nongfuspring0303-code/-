# 收敛执行计划_任务总书 v3

---

## 一、项目背景与总目标

本计划旨在完成从"事件认知"到"交易动作"的最后一公里集成。所有工作的终点是输出符合《附录 B》规范的 **ActionCard JSON**。

**核心原则**：
1. LLM 负责理解（提供证据与要素），规则系统负责裁决（路径收敛与动作决策）
2. NO A1 → NO TRADE（必须硬编码，不止注释）
3. 事件状态到交易状态必须有固定映射（附录 F）
4. 治理层异步，不阻主战链时延

---

## 二、角色分工与核心 KPI

### 成员 A：AI 语义理解层
**负责文档**：《LLM 新闻分析层设计与字段映射规范 v2.1》

**核心任务**：
- 重写 Prompt，强制输出结构化事件要素（不含路径裁决）
- 升级 `ai_signal_adapter.py`，废弃 4 字段旧逻辑

**交付物**：能输出以下字段的 `EventObject`：
- `event_id` / `event_type` / `event_time` / `a0_event_strength` / `expectation_gap`
- `event_state`（Initial / Developing / Peak / Fading / Dead）
- `transmission_candidates`（候选传导要素，≤3条，不做主路径裁决）
- `risk_flags` / `evidence_spans`

**KPI（可验收）**：
- 基线样本：团队共同认定的 10 条黄金新闻样本
- 统计窗口：提交前全量通过
- 计算公式：Schema 校验通过率 = 通过条数 / 10 ≥ 90%；`event_state` 与人工标注一致率 ≥ 80%

---

### 成员 B：市场验证与路径裁决层
**负责文档**：《附录 C 路径裁决规则规范》、《附录 D 市场确认规范》

**核心任务**：
- 实现三层市场确认（宏观 / 板块 / 龙头）
- 实现路径收敛（1 主 + 0~2 次，保留强 P2 白名单）

**交付物 1：`PathDecision`**（路径裁决结果）
- `primary_path`（唯一主路径 + 六维评分）
- `secondary_paths`（0-2 条强 P2 路径）
- `rejected_paths`（带放弃原因）
- `target_etf` / `target_leader` / `target_sector`

**交付物 2：`MarketStatus`**（市场确认结果）
- `macro_confirmation`（supportive / neutral / hostile）
- `sector_confirmation`（strong / medium / weak / none）
- `leader_confirmation`（confirmed / partial / unconfirmed / failed）
- `a1_market_validation`（pass / partial / fail）
- `positive_signals` / `negative_signals` / `reason_text`

**KPI（可验收）**：
- 基线样本：同上 10 条黄金样本 + 5 条"逻辑成立但市场不认"的对比样本
- A1 fail 拦截率 = 实际拦截条数 / 预期应拦截条数 ≥ 100%（硬约束，不允许漏判）

---

### 成员 C / 系统集成：主链编排与输出层
**负责文档**：《附录 B 交易动作输出卡片规范》、《附录 F 事件状态到交易状态映射规范》

**核心任务**：重构 `workflow_runner.py`，聚合 A+B 输出，生成完整 ActionCard

**交付物：`ActionCard`（六层完整结构）**

| 层级 | 核心字段 |
|------|---------|
| 事件层 | `event_name` / `event_type` / `event_time` / `time_window` / `evidence_grade` / `catalyst_state` |
| 主路径层 | `primary_path` / `secondary_paths` / `rejected_paths` / `target_bucket` |
| 市场确认层 | `macro_state` / `sector_confirmation` / `leader_confirmation` / `a1_market_validation` |
| 动作层 | `trading_state` / `trade_grade`(A/B/C/D) / `trade_decision` / `best_target` / `best_setup` / `position_tier` / `execution_window` |
| 风险层 | `risk_switches` / `invalidators` / `downgrade_rules` |
| 结论层 | `one_line_verdict` |

**硬约束（必须代码实现，不止注释）**：
- `a1_market_validation == fail` → `trade_decision ∈ {observe_only, avoid}`，`position_tier = none`
- `catalyst_state == Dead` → `trade_decision = avoid`
- `evidence_grade == C` → 禁止生成 `tradable` 或 `overnight_allowed`

## 三、并行开发协议

```
成员 A ─── EventObject ──┐
                          ├──→ workflow_runner.py → ActionCard
成员 B ─── MarketStatus ─┘
           PathDecision ──┘
```

1. 第 1 天：三方确认三个 Dataclass 接口（EventObject / PathDecision / MarketStatus）
2. 第 2-3 天：A/B 并行开发，C 搭建 Runner 框架
3. 第 4 天：联调，跑 10 条黄金样本 E2E 验收

---

## 四、禁止事项

- LLM 不得输出最终主路径（只输出候选要素）
- 各层不得自由命名枚举值，必须严格遵循附录定义
- 治理层落盘不得同步阻塞 Runner 的返回路径
