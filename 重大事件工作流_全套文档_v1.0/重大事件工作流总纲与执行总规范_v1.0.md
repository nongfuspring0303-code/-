## 一、执行摘要

重大事件工作流不是新闻抓取系统，也不是资讯推送工具。  
本项目的核心目标是：

> **把重大外部事件，尽可能快地压缩成可执行的交易判断与交易动作。**

系统当前已经具备事件识别、证据分级、因果传导、资产映射、路径裁决及候选输出等高级雏形，  
但当前状态仍更接近一个“高级事件认知系统”，尚未完全进化成“高胜率交易执行中枢”。

当前最大短板不在功能数量，而在最后一公里：

- 主路径收敛不够狠
- 市场承接验证不够强
- 输出动作化程度不够高
- 系统复杂度存在隐形延迟税

因此，当前阶段总目标不是继续横向扩模块，  
而是纵向压缩主路径、主对象、主动作与主风险开关，  
让系统更快、更狠、更可执行。

## 二、项目一句话定义与边界

### 2.1 项目一句话定义

重大事件工作流是一个面向美股事件驱动交易的：

**事件识别 → 证据验证 → 因果传导 → 资产映射 → 路径裁决 → 交易输出**

一体化决策系统。

### 2.2 项目不是什么

本项目不是：

- 新闻聚合器
- 资讯摘要系统
- 单纯解释逻辑的分析系统
- 直接替代风控与执行的 AI 决策黑箱

### 2.3 项目最终成品

项目最终成品不是“更多新闻”或“更长解释”，而是：

- 能不能做
- 做谁
- 怎么做
- 做到什么程度
- 什么条件下撤退

### 2.4 非适用范围

本总纲不直接展开以下内容：

- 具体代码实现细节
- 前端展示样式
- 数据库 / 缓存 / 消息队列选型
- 券商执行接入层细节

这些内容由专项规范或实现文档负责。

## 三、系统主链路总览

### 3.1 主链路

**事件采集 → 证据校验 → 事件分类 → 资产映射 → 路径裁决 → 市场确认 → 交易输出 → 回放归因**

### 3.2 各层作用

- **事件采集**：解决“有没有事件”
- **证据校验**：解决“能不能信”
- **事件分类**：解决“它是什么类型的催化”
- **资产映射**：解决“影响谁”
- **路径裁决**：解决“先做谁、为什么先做它”
- **市场确认**：解决“市场是否真实承接”
- **交易输出**：解决“能不能做、怎么做”
- **回放归因**：解决“系统如何自我优化”

### 3.3 主战链与治理链

#### 主战链
事件采集 → 证据校验 → 资产映射 → 路径裁决 → 市场确认 → 交易输出

#### 治理链
审计留痕 → 回放归因 → 预算管理 → Prompt / Schema 治理 → 版本追踪

## 四、系统角色与职责边界

### 4.1 系统级边界

- 新闻层负责输入事件
- 理解层负责结构化理解事件
- 规则层负责验证、裁决、降级与执行门禁
- 输出层负责生成交易动作卡片
- 治理层负责审计、预算、回放与版本追踪

### 4.2 LLM 与规则系统边界

#### LLM 负责
- 理解语义与逻辑
- 输出结构化事件对象
- 提供预期差、叙事阶段、传导路径等辅助理解结果

#### 规则系统负责
- 决定事件能否进入主链
- 决定是否满足市场确认
- 决定是否允许交易
- 决定风险闸门是否放行
- 决定降级、拦截与回退策略

### 4.3 明确红线

- LLM 不得直接输出交易指令
- LLM 不得直接给出仓位比例
- A1 市场验证必须由真实价格与成交量完成
- AI 可辅助，不可替代规则系统与硬风控

### 4.4 各层成员职责映射

- **上游输入**：事件进入速度、来源可信度、首发与回退控制
- **中游理解**：事件分类、证据稳定性、传导路径收敛
- **下游输出**：交易动作清晰化
- **治理层**：门禁、预算、验证、审计、发布治理

## 五、总纲层固定原则

### 5.1 交易优先原则

**新闻只是输入，交易动作才是成品。**

### 5.2 主路径优先原则

不是相关资产越多越好，而是主路径越短、越硬、越有承接越好。

### 5.3 市场确认原则

不是逻辑成立就能做，必须经过市场承接验证。

### 5.4 复杂度约束原则

不是制度越重越好，主链速度同样是交易价值。  
凡增加主链延迟税的设计，必须证明其收益显著高于成本。

### 5.5 新增设计准入原则

所有新增设计都必须证明自己能提高以下质量，否则不进主链：

- 做 / 不做
- 做谁
- 怎么做
- 何时退出

## 六、LLM 职责与接口关系（总纲摘要层）

> 本章仅保留 LLM 层在总纲中“必须被全员理解”的摘要信息。  
> 完整字段、Schema、Prompt、预算与治理细节，见附录 A。

### 6.1 LLM 的系统角色

LLM 是受约束的事件理解模块，不是最终交易裁决器。  
它负责将新闻文本转为系统可消费的 **Event Object**。

### 6.2 LLM 输出三层结构

#### 抽取层（Facts）
- 事件类型
- 实体清单
- 极简情绪判定

#### 推理层（Inference）
- 事件强度
- 预期差
- 叙事 / 事实属性
- 传导路径
- 波及范围
- 叙事阶段
- 证据片段

#### 治理层（Governance）
- 模型版本
- Prompt 版本
- 输入快照
- 延迟数据
- 审计追溯字段

### 6.3 LLM 与主链的接点

- 接入 **A0**：事件识别与事件对象构建
- 接入 **A-1**：预期差辅助理解
- 接入 **A2**：因果传导链辅助
- 接入 **A1**：只能做软验证辅助，不得替代真实市场验证
- 接入 **风险层**：可作为语义型 Risk Guardian 的补充层

### 6.4 总纲级红线

- LLM 不输出 Action
- LLM 不输出 Position Size
- A1 必须来自真实价格与成交量
- 预算耗尽或延迟过高时，必须允许无缝 fallback

## 七、当前阶段判断与系统瓶颈

### 7.1 当前阶段判断

系统当前已具备较强的高级事件认知能力，  
但尚未完全进化为高胜率交易执行中枢。

### 7.2 当前最强项

- 事件理解能力
- 证据分级意识
- 路径裁决意识
- 工程纪律与审计意识

### 7.3 当前最弱项

- 交易动作压缩不足
- 主路径收敛力度不足
- 市场承接验证不足
- 收益归因与自我优化能力不足

### 7.4 当前核心风险

- 速度风险
- 过度确认风险
- 路径膨胀风险
- 市场错配风险
- 执行漂移风险

---

## 八、当前阶段执行优先级

### 8.1 P0（最高优先级）

1. **统一交易动作输出卡片**
2. **建立一级路径优先机制**
3. **内嵌市场确认闸门**

### 8.2 P1

1. 建立观察级 / 试仓级 / 可做级 / 加仓级渐进式动作机制
2. 完善事件状态到交易状态映射

### 8.3 P2

1. 建立收益归因日志体系
2. 建立事件类型 / 证据组合 / 时间窗口有效性统计

### 8.4 当前不优先事项

- 纯展示层修辞优化
- 不影响主动作质量的字段扩写
- 只增加可解释性、不增加交易质量的复杂逻辑

## 九、团队执行规则

### 9.1 新伙伴进入项目的第一原则

先理解自己处于哪一层，再决定改什么，  
不允许一上来全链乱改。

### 9.2 任何改动都必须回答 5 个问题

1. 你的输入是什么？
2. 你的输出是什么？
3. 你影响主链哪一步？
4. 你提高了哪个交易质量指标？
5. 你引入了哪些延迟或复杂度成本？

### 9.3 主链准入规则

任何模块若不能显著提高“做 / 不做 / 做谁 / 怎么做”的质量，  
不进入主链。

### 9.4 团队统一口径

- 我们不是在做资讯系统，我们在做事件驱动交易判断系统
- 新闻只是输入，交易动作才是成品
- 主路径短于路径多
- 市场承接高于单纯逻辑成立
- 速度是交易价值的一部分
