## 1. 项目目标与边界

### 1.1 项目目标

- 建立重大事件驱动的“情报 A 层 -> 分析 B 层 -> 执行 C 层”本地可联调系统。
- 提供前端监控与配置页面，支持实时事件、板块、机会卡可视化。
- 建立可复核的质量体系：单测、门禁脚本、健康检查、证据落盘。
- 提供 dev/prod 角色隔离与 master/worker 主链转发机制。

### 1.2 非目标（当前阶段）

- 非“交易所级毫秒低延迟执行系统”。
- 非“机构付费终端同级”新闻能力（Reuters/Bloomberg 专线级）。
- 非“完全自动化无人值守生产交易”系统。

## 4. 端到端全链路时序

### 4.1 主流程（新闻驱动）

1. `RealtimeNewsMonitor` 轮询新闻源（`DataAdapter.fetch_news()`）
2. 命中新闻后触发 `EventCapture`
3. 构建 A/B 输入并执行 `FullWorkflowRunner`
4. A 层：`IntelPipeline`（EventCapture/SourceRanker/Severity/EventObjectifier）
5. B 层：Lifecycle/Fatigue/Conduction/MarketValidation/Signal/Opportunity
6. C 层 API 依次接收：
   - `event_update`
   - `sector_update`
   - `opportunity_update`
7. EventBus 广播并写历史（可配置落盘）
8. Canvas 通过 WS/API 展示实时卡片

### 4.2 关键实现入口

- 实时触发：`scripts/realtime_news_monitor.py`
- 全链路编排：`scripts/full_workflow_runner.py`
- C 栈启动：`scripts/run_c_module_stack.py`
- EventBus：`scripts/event_bus.py`

## 5. 核心数据对象与事件契约

### 5.1 三类推送对象

- `event_update`：事件卡（标题、来源、严重度、新闻时间/推送时间）
- `sector_update`：板块热力（方向、置信度、影响分）
- `opportunity_update`：股票机会（symbol、signal、entry_zone、risk_flags）
- 命名约定说明：对象字段/类型使用下划线（如 `event_type`, `event_update`），HTTP ingest 路径使用 kebab-case（如 `/api/ingest/event-update`），两者语义一一对应。

Schema 文件：

- `schemas/event_update.yaml`
- `schemas/sector_update.yaml`
- `schemas/opportunity_update.yaml`

### 5.2 关键 ID 与追踪字段

- `trace_id`：主追踪链路 ID
- `request_id` / `batch_id`：批次与请求归属
- `schema_version`：当前统一为 `v1.0`
- `event_id`：标准化事件主键（由 Intel 层输出）

## 7. 核心脚本分层说明

### 7.1 A 层：情报标准化

- `intel_modules.py`
  - `EventCapture`：触发检测
  - `SourceRanker`：来源等级 A/B/C
  - `SeverityEstimator`：E0-E4
  - `EventObjectifier`：标准事件对象

- `ai_event_intel.py`
  - `NewsIngestion`：RSS/Atom 拉取、标准化、去重
  - `EventEvidenceScorer`：证据分与新鲜度打分

### 7.2 B 层：传导与机会

- `conduction_mapper.py`
  - 基于类别 + 模板 + 语义选择链路
  - 输出 macro/sector/stock_candidates

- `opportunity_score.py`
  - 板块到股票机会构建
  - 价格来源优先级、风险标记
  - 板块别名字典归一（如 `Energy -> 能源`）

- `signal_scorer.py` / `market_validator.py` / `fatigue_calculator.py`
  - 信号评分、市场验证、叙事疲劳

### 7.3 C 层：推送与可视化

- `run_c_module_stack.py`：WS + API + WEB 一键联调

- `config_api_server.py`：ingest/feedback/config API
- `realtime_news_monitor.py`：将 A/B 结果推送 C 层

## 8. 配置体系详解

核心文件：`configs/edt-modules-config.yaml`

### 8.1 runtime（全局运行）

- `role`: `dev|prod`
- `node_role`: `master|worker`
- `master_api`: worker 转发目标
- `strict_fallback`: 是否严格禁回退
- `price_fetch`: 实时价格拉取开关/TTL
- `semantic.enabled`: 语义分析总开关（禁用时退回纯规则）
- `semantic.provider` / `semantic.model`: 语义提供方与模型标识（当前：glm-4.7-flash）
- `semantic.min_confidence`: 语义命中最小置信度阈值（低于阈值自动 abstain）
- `semantic.timeout_ms`: 单次语义调用超时阈值
- `semantic.full_enable`: 全量启用开关（灰度期可配合 `enabled` 控制）
- `semantic.emergency_disable`: 应急熔断开关（运行中一键退回规则模式）

> **2026-04-10 更新**：语义分析现已集成 GLM-4.7 Flash API 调用（通过 ZAI API），遇 rate limit 时自动降级到 deterministic keyword fallback。

### 8.2 NewsIngestion 源

当前 sources 包含 Reuters/Google/CNBC/SEC/Fed 等多个 RSS/Atom 源。

### 8.3 canary_source_health（新增实时证据门禁）

- 支持多源回退链（sina -> reuters -> google reuters）
- **主源**: Sina 直播 API (`http://zhibo.sina.com.cn/api/zhibo/feed`)
- 指标窗口（5m/30m/1h）
- 门禁阈值：成功率 >= 95%, p95 延迟 <= 3000ms, 新鲜度 lag <= 21600s, 30m 新增条数 >= 1
- 结果落盘：`logs/canary_live/`
- 验证命令：
  ```bash
  python3 scripts/verify_canary_source_health.py --refresh
  # 可选（待全量门禁开启）
  python3 scripts/verify_news_freshness.py
  ```

## 13. 已知限制与当前风险

### 13.1 新闻实时性现状

- 直连 Reuters 源在当前网络环境下可能出现 401/404/反爬拦截。
- canary 可通过回退源（如 Google Reuters RSS）维持“可用性证据”，但不代表秒级新闻源。
- NewsAPI key 缺失时会回退，不应将回退 GREEN 误解为“高实时达标”。

### 13.2 worker 转发严谨性风险

- `realtime_news_monitor.py` 的 worker 分支仍需重点复核 payload 契约一致性与 endpoint 对应关系。

### 13.3 外部行情源风险

- Yahoo 部分接口可能出现 401，导致实时价格不可用并触发 WATCH 降级。

## 15. 常见故障排查手册

### 15.1 页面有数据但不更新

- 检查 `run_local.sh` 是否启动成功
- 检查 `realtime_news_monitor.py` 是否在跑
- 检查 API 端口（默认 `18787`）和 Web 端口（默认 `18080`）

### 15.2 新闻时间与推送时间异常

- 检查 `event_update.news_timestamp` 是否存在
- 校验来源字段优先级：`news_timestamp/published_at/detected_at/timestamp`

### 15.3 机会卡空白

- 检查 `sector_update` 和 `opportunity_update` 是否同时入总线
- 检查板块别名映射（`configs/sector_aliases.yaml`）

### 15.4 canary 显示 GREEN 但实时性差

- 查看是否命中了 `source_overrides` 放宽阈值
- 查看 `attempted_sources`，确认是否回退到聚合源

### 15.5 AI 语义触发异常（应急回滚 Runbook）

- 适用场景：`realtime_news_monitor.py` 出现语义误触发、超时激增、误报率突增。
- 第一步（立即止血）：在 `configs/edt-modules-config.yaml` 设置 `runtime.semantic.emergency_disable: true`，必要时同步设 `runtime.semantic.enabled: false`。
- 第二步（重启生效）：重启运行栈（`./run_local.sh` + 新闻进程），确认新事件 `ai_verdict` 退回规则路径。
- 第三步（健康确认）：执行 `python3 scripts/system_healthcheck.py --mode dev` 与目标回归用例，确保主链可用后再排查语义侧根因。
- 恢复策略：问题定位后先小流量恢复 `emergency_disable: false`，观察 canary 与误报指标，再恢复全量。
